# CLAUDE.md

## Цель проекта

Собрать учебное веб-приложение «Запись на звонок» — упрощенный сервис бронирования времени по мотивам Cal.com.

Сервис должен позволять:

* владельцу календаря создавать типы событий;
* гостю просматривать доступные типы событий;
* гостю выбирать тип события, дату и свободный слот;
* гостю создавать бронирование без регистрации и авторизации;
* владельцу просматривать список предстоящих встреч в админке.

В проекте **нет**:

* регистрации;
* авторизации;
* личных кабинетов;
* интеграций с внешними календарями;
* сложных ролей и прав доступа.

## Роли в системе

### Владелец календаря

Это один заранее заданный профиль без механики входа в систему.

Владелец календаря может:

* создавать типы событий;
* задавать для типа события:

  * `id`;
  * `name`;
  * `description`;
  * `durationMinutes`;
* просматривать страницу предстоящих встреч;
* видеть в одном списке бронирования всех типов событий.

### Гость

Гость работает с публичной частью приложения без аккаунта и без входа в систему.

Гость может:

* открыть страницу с типами событий;
* видеть для каждого типа события название, описание и длительность;
* выбрать тип события;
* открыть календарь;
* выбрать свободный слот;
* создать бронирование на выбранный слот.

## API-first подход

Проект ведем по модели **API First**:

1. Сначала фиксируем поведение системы.
2. Затем описываем API-контракт.
3. Только после этого переходим к реализации frontend и backend.

Для описания контракта используем **TypeSpec**.

## TypeSpec

TypeSpec используется как основной инструмент для проектирования API.

Что важно учитывать:

* TypeSpec нужен до начала реализации;
* TypeSpec является основным источником истины для API-контракта и OpenAPI-спецификации;
* контракт, описанный в TypeSpec, должен быть согласован до начала разработки frontend и backend;
* TypeSpec визуально похож на TypeScript, но не является полноценным языком программирования;
* при изменении API сначала обновляется TypeSpec-спецификация, затем реализация;
* отдельный markdown-документ по API создается только при необходимости, если нужно дополнительно пояснить бизнес-правила или договоренности.

## Принципы работы

1. Сначала фиксируем поведение системы и API-контракт.
2. Затем реализуем интерфейс поверх согласованного контракта.
3. После этого реализуем backend.
4. Дальше покрываем ключевые сценарии тестами.
5. В конце готовим приложение к деплою в Docker.

## Основной пользовательский сценарий

### Публичный сценарий записи

1. Пользователь открывает лендинг.
2. Нажимает кнопку записи.
3. Видит список типов событий.
4. Выбирает тип события.
5. Видит календарь и доступные временные слоты.
6. Выбирает свободный слот.
7. Заполняет форму записи.
8. Подтверждает бронирование.
9. Видит подтверждение успешной записи.

### Сценарий владельца календаря

1. Владелец открывает админку.
2. Видит список предстоящих встреч.
3. Может просматривать основные данные записи.

## Черновой функциональный объем

### Обязательно

* публичная страница с описанием сервиса;
* страница выбора типа события;
* отображение для каждого типа события:

  * названия;
  * описания;
  * длительности;
* страница выбора даты и свободного слота;
* форма бронирования;
* создание записи на сервере;
* страница предстоящих встреч для владельца календаря;
* создание типов событий владельцем календаря;
* защита от бронирования уже занятого слота;
* правило глобальной занятости слота: на одно и то же время нельзя создать две записи, даже если это разные типы событий;
* Docker-сборка и запуск приложения в контейнере.

### Допустимые упрощения

* один владелец календаря;
* владелец календаря заранее задан в системе;
* гость не имеет аккаунта;
* хранение в SQLite или другой простой БД;
* простая админка без авторизации.

## Почему выбран именно этот стек

* **Python / FastAPI**: основной backend пишем на знакомом стеке, чтобы быстрее проверять логику, API и тесты.
* **React + TypeScript**: удобно собирать интерфейс по референсам и последовательно реализовывать экранные состояния.
* **SQLite**: достаточно для MVP без лишней инфраструктуры.
* **Playwright**: хорошо подходит для проверки ключевых пользовательских сценариев на уровне интерфейса.
* **Docker**: обязателен по условиям проекта и нужен для воспроизводимого запуска.
